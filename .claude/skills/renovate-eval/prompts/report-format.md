# Report Format

This documents the rendered markdown report format. The report is generated by a
template from structured JSON data (eval-data.json) — the evaluator does NOT
write the report directly.

## Report Structure

Every factual claim must have a full markdown link `[text](url)` — never bare
`#123` or unlinked references.

Sections are omitted from the rendered output when the corresponding JSON field
is `null`. The Hazards & Risks section is always present — the evaluator writes
"None identified" with a brief justification if there are no hazards.

```markdown
# [Package] ([type]) [OldVersion] -> [NewVersion]

**Risk:** [emoji] [Label]

## The Deep Dive

### Update Scope

What is updating and what the version deltas are. Explicitly state unchanged
components.

### Performance & Stability

Performance improvements, stability fixes, resource usage changes. Link each
item to its PR or issue.

### Features & UX

For EACH new feature, state: (1) what it does, (2) whether the user's config
uses it, (3) how to enable it with specific config keys/commands. "Feature X
added" without enablement guidance is insufficient. Link each item to its PR or
issue.

### Security

CVE IDs with CVSS scores as full markdown links, whether user is affected based
on their config.

### Key Fixes

Cross-reference bug fixes against actual config and usage patterns. Link each
item to its PR or issue.

### Newer Versions

Check whether newer versions exist beyond what this PR proposes. If they do,
evaluate their changelogs for:

- Bugs or regressions INTRODUCED in the proposed version range (not present in
  the current version) and fixed in a later release. These MUST be called out if
  relevant to the deployment's config — they mean the proposed version has known
  issues. This should influence the verdict toward Caution or Risk.
- Security vulnerabilities INTRODUCED in the proposed version range (not present
  in the current version) and fixed in a later release. Always flag these
  regardless of config relevance. This should influence the verdict toward Risk.
- Pre-existing issues (bugs, CVEs, or vulnerabilities present in BOTH the
  current and proposed versions) do NOT affect the verdict — they are not risks
  introduced by this PR. Note them as context if serious, but they must not
  drive the label. A CVE that exists in both versions is not a reason to flag
  the update as risky.

## Hazards & Risks

REQUIRED even if no risks exist. List every breaking change, deprecation, and
migration with deployment-specific impact assessment. If there are genuinely no
hazards, write "None identified" with a brief explanation of why (e.g., "no
breaking changes in this release").

## Sources

- [Label](url) for each source consulted

---

## [emoji] Verdict: [Label]

[1-2 sentence rationale, plus post-merge follow-up actions if any]
```

## Multi-Package PRs

The title lists all packages, comma-separated with types:
`# PostgreSQL (docker) 16.2 -> 16.3, Sonarr (helm) 2.10.5 -> 2.10.6`

Duplicate packages (same name, type, old_version, new_version) are listed only
once. Use per-package sub-sections under each topic heading. The verdict uses
highest-risk-wins for the label.

## Verdict Mapping

- 🟢 **Safe** (`renovate:safe`) — no concerns, straightforward update. Security
  patches that fix CVEs without introducing behavioral changes are safe — the
  change reduces risk without adding any.
- 🟡 **Caution** (`renovate:caution`) — behavioral changes worth validating
- 🟠 **Breaking** (`renovate:breaking`) — breaking changes, needs config rework
- 🔴 **Risk** (`renovate:risk`) — known issues, regressions, or low confidence

Use Unicode emoji (not shortcodes like `:green_circle:`).

## CI Status

When rendered for local display, the status line includes CI:
`**Risk:** 🟢 Safe | **CI:** passing`

When rendered for posting as a GitHub comment, CI status is omitted from the
status line (it goes stale). Live CI status is available via the `status`
subcommand.

## Failed Audit Banner

If the report did not pass audit after 3 rounds, this banner is prepended (by
the evaluation script, not the evaluator):

```markdown
> ⚠️ **This report did not pass automated quality review.** Treat with
> skepticism.
```
